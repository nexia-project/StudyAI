A) NOTEBOOK (RAG) — CLONE DO NOTEBOOKLM (INTEGRADO)
========================================================

Objetivo:
- Cadernos (notebooks) com fontes (PDF, DOCX, links, YouTube, áudio, imagens etc.)
- Chat que responde EXCLUSIVAMENTE com base nas fontes (RAG) com citações clicáveis
- Studio com geração de conteúdo (resumos, flashcards, quizzes, mapa mental, podcast, etc.)
- Histórico de conversas e persona por caderno

A1) Tela inicial (listagem de cadernos)
- Grid/lista de cadernos com: título, data, nº de fontes, última atualização
- Buscar e ordenar
- CTA “Criar novo caderno”

A2) Página de um caderno (layout responsivo)
Desktop: 3 painéis
- Esquerda: Fontes
- Centro: Chat
- Direita: Studio
Tablet: Studio pode virar drawer
Mobile: tabs “Fontes | Chat | Studio”

A3) Fontes (painel esquerdo)
Suportar import/upload:
- PDF, DOCX, TXT/MD
- Links web (HTML → texto limpo)
- YouTube (transcrição com timestamps)
- Áudio (Whisper)
- Imagens (OCR)
- CSV/XLSX, PPTX, Google Docs/Sheets (link público), EPUB

Cada fonte: status (processando/pronto/erro), preview, remover.
Drag & drop, upload múltiplo.

A4) Pipeline RAG (backend)
Ao adicionar fonte:
1) Extração de texto conforme tipo
2) Chunking ~500 tokens overlap ~50 + metadados (sourceId, pageNumber/pos)
3) Embeddings (preferir o padrão já usado no projeto; senão OpenAI text-embedding-3-small 1536)
4) Armazenar com busca vetorial (pgvector ou equivalente do projeto)

Ao perguntar:
1) Embedding da pergunta
2) Busca top 10–15 chunks relevantes no caderno
3) Montar contexto com identificação da fonte/página
4) Chamar modelo padrão do projeto (ou GPT-4o), com system prompt:
   “Responda EXCLUSIVAMENTE com base no contexto; se não houver, diga que não encontrou; cite [1],[2]…”
5) Resposta em streaming

A5) Chat (painel central)
- Markdown
- Citações clicáveis [1],[2] abrindo tooltip/modal com trecho + fonte + página + “abrir fonte”
- Histórico de conversas por caderno
- “Nova conversa”
- Sugestões iniciais

A6) Persona do caderno
Campo “Instrução personalizada” (até ~5000 chars) injetado no system prompt.

A7) Studio (painel direito)
Botões (em fases):
- Resumo (fase 1/2)
- Guia de estudo (fase 2)
- Flashcards com repetição espaçada (fase 2)
- Quiz (fase 2)
- Mapa mental (fase 2/3)
- Audio Overview (Podcast) (fase 3)
- Timeline / Tabela / Slides / Relatório (fase 3/4)

Audio Overview:
- Gerar script com 2 hosts
- TTS com duas vozes (ou provider existente)
- Player: play/pause, barra, velocidade, pular 10s, download
- Modo “participar”: pausa, usuário pergunta, hosts respondem em áudio

A8) Banco de dados (RAG)
Criar tabelas novas (sem duplicar User etc.; seguir naming do projeto, ex. prefixo rag_):
- rag_notebooks, rag_sources, rag_chunks, rag_conversations, rag_messages
- rag_flashcard_sets, rag_flashcards, rag_flashcard_reviews
- rag_quizzes, rag_quiz_attempts
- rag_studio_outputs, rag_notebook_shares, rag_deep_research_jobs
Criar índices apropriados (incluindo índice vetorial).

A9) Premium gating
Features caras (Deep Research, Audio Overview etc.) podem ser Premium (usar sistema Premium já existente).

========================================================

Objetivo: professor consegue planejar, criar, aplicar, corrigir e acompanhar.

B1) Dashboard do Professor
- Cards “Minhas Turmas”: nome, série/ano, nº alunos, engajamento, próxima atividade
- Alertas: alunos em risco, pendências de correção, provas próximas

B2) Gestão de Turmas

B7) Acompanhamento e relatórios

========================================================
D) SISTEMA DE COLETA DE DADOS À PROVA DE FALHAS (OBRIGATÓRIO)
========================================================

Problema: dashboards e gráficos vazios/inconsistentes.
Solução: camada unificada de eventos + métricas consolidadas.

D1) Tabela activity_events (event log)
Campos:
- userId, eventType, entityType, entityId, classId?, notebookId?, metadata JSON, createdAt
Índices por (userId,eventType,createdAt), (classId,eventType,createdAt), etc.

Registrar eventos para:
- login/sessão
- plano de estudos
- quiz (start/answer/complete)
- flashcards (review)
- redação (submitted/corrected)
- notebook RAG (source added/chat/studio)
- ações do professor (aula criada, atividade atribuída, nota dada)

D2) Serviço central trackEvent()
- Validação: nunca evento incompleto
- “Fire and forget”: falha não pode quebrar a UX
- Logs de erro
- Chamar em TODOS os fluxos relevantes

D3) Tabela daily_metrics (consolidação diária, upsert)
- Por userId + date (e classId opcional)
- totalStudyMinutes, quizzesCompleted, quizzesAvgScore, flashcardsReviewed, topicsCompleted,
  notebookChats, notebookSourcesAdded, essaysSubmitted, loginCount, lastActiveAt etc.
- Job de atualização:
  - por cron (15 min) ou debounced quando evento entra
- Dashboards NUNCA fazem queries complexas direto nas tabelas brutas; lêem daily_metrics.

D4) Tratamento de “sem dados”
- APIs retornam objeto completo + hasData flag
- Frontend tem 3 estados: loading / vazio / com dados
- Nunca mostrar NaN/undefined/null; “—” para ausência de dados

D5) Validação E2E + auto-correção
- Scripts diários que comparam tabelas brutas (ex quizzes) vs eventos
- Se divergência, logar alerta e (se possível) gerar eventos faltantes

D6) Seed de dados para testar dashboards
- Script que cria 3 turmas, 15–25 alunos, 30 dias de eventos
- Perfis: engajados/medianos/em risco
- Permite validar gráficos antes de dados reais
